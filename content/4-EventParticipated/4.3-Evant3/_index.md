# Reflection Report: “Level Up Your AWS Skills with Cloud Quest and Floci”

### Purpose of the Event

- To understand how beginners can learn AWS more effectively through hands-on practice, gamified learning, and guided quests.
- To identify common challenges faced by new AWS learners, especially concerns about billing, undeleted resources, and unexpected extra charges.
- To explore **AWS Cloud Quest** as an immersive learning environment that helps learners understand cloud architecture and AWS services through practical missions.
- To learn about **Floci**, an open-source AWS service emulator that supports local architecture testing without creating real cloud resources at the early learning stage.
- To distinguish between learning theory, practicing in a simulated local environment, and deploying to real AWS infrastructure.
- To listen to real experiences from hackathon participants and understand how a prototype or MVP can be built under time pressure.
- To understand the role of confidence in learning, career development, and personal growth.
- To observe a practical example of migrating from a current technology stack to an AWS-based architecture, thereby gaining a clearer view of cloud architecture and DevOps thinking.

### Key Event Highlights

#### Level Up Your AWS Skills with Cloud Quest and Floci

The main session was titled **“Level Up Your AWS Skills with Cloud Quest and Floci”**, presented by **Huỳnh Thái Linh** as part of the First Cloud AI Journey event. The session focused on how students and beginners can approach AWS learning in a more practical, accessible, and cost-conscious way.

The main points of this session included:

- Common challenges when learning AWS through hands-on practice.
- An introduction to AWS Cloud Quest.
- An introduction to the open-source tool Floci.
- How to combine Cloud Quest and Floci to learn AWS more effectively.

#### Challenges for Beginners Learning AWS

One major issue when learning AWS is the concern about cost. When practicing directly on the real cloud, learners may accidentally create resources incorrectly, forget to delete resources, or misunderstand the billing model. These mistakes can result in unexpected charges.

The concerns mentioned in the presentation included:

- Checking the AWS bill repeatedly.
- Forgetting to delete cloud resources after practice.
- Worrying about additional charges.
- Lacking confidence when testing cloud architectures because of fear of wrong configuration.

A key point is that the difficulty of learning AWS is not only technical. It is also psychological. Beginners are often afraid of making mistakes and losing money. Therefore, an effective learning roadmap should help learners understand cloud architecture while also providing a safer environment for practice.

#### AWS Cloud Quest

**AWS Cloud Quest** was introduced as a gamified AWS learning platform. Instead of only reading documentation or watching videos, learners complete missions that simulate real-world scenarios. This helps them build cloud skills according to specific roles.

Main highlights:

- Learning AWS through games and hands-on quests.
- Step-by-step guidance that is suitable for beginners.
- Helping learners understand architectural thinking before going deeper into code.
- Providing badges for roles such as Cloud Practitioner, Solutions Architect, Serverless Developer, Machine Learning, and Security.
- Creating a more active learning experience because learners must solve problems instead of only receiving theory.

From this session, I realized that Cloud Quest is useful for building a foundation in cloud architecture. Beginners can visualize how AWS services work together before dealing with more complex configurations in a real production environment.

#### Open Source Floci

The next part introduced **Floci**, a tool that emulates AWS services in a local environment. Floci helps learners and developers quickly test architecture or AWS integration logic without immediately deploying to the real cloud.

Key values of Floci include:

- Simulating several AWS services locally.
- Avoiding cloud costs during early experimentation.
- Supporting fast architecture testing.
- Being suitable for learning, demos, prototypes, and idea validation.
- Reducing the fear of making mistakes on a real AWS account.

Floci is especially useful when learners want to write code, test APIs, validate workflows, or simulate part of a cloud system without creating real AWS resources.

#### Comparison Between Floci and LocalStack

In the presentation, Floci was compared with **LocalStack** across several criteria. The comparison included:

- Speed and performance.
- Memory and storage usage.
- Service support and compatibility.
- Licensing and pricing.

According to the slide, Floci was emphasized as lightweight, fast, and suitable for quick testing. However, it is important to recognize that a local emulator cannot fully replace the real AWS environment. It should be treated as a supporting layer during learning and development.

#### Limitations of Floci

One important part of the session discussed the limitations of Floci. This is significant because learners may mistakenly assume that an emulator can completely replace real cloud infrastructure.

The limitations mentioned included:

- Limited AWS service coverage.
- Some services may use fake or mock results.
- It is not fully identical to real AWS.
- It cannot completely replace real AWS testing.

This means Floci is most suitable for learning, coding, quick testing, and prototyping. When a system is prepared for production, it must still be deployed and tested on real AWS to evaluate performance, security, cost, and reliability accurately.

#### Practical AWS Learning Roadmap

One useful part of the event was the practical learning roadmap, which consisted of three phases:

| Phase | Content | Suitable Tool |
| --- | --- | --- |
| Phase 1 | Build architectural thinking and understand how AWS services work together | AWS Cloud Quest |
| Phase 2 | Write code, test quickly, and experiment with architecture | Open-Source Floci |
| Phase 3 | Deploy and operate in a real production environment | Real AWS |

This roadmap is reasonable because it does not force beginners to use the real cloud too early. First, learners need to understand architecture. Then they can move to coding and local testing. Finally, they can deploy on real AWS and deal with practical issues such as cost, security, logging, monitoring, and operations.

### Hackathon and Product-Building Mindset

Another part of the event focused on **Hackathon**, presented by **The Ballers**, including Huỳnh An Khuong, Mai Quoc Anh, and Nguyen Tran Minh Quan. This section explained what a hackathon is, why students should participate, and what the real experience of building a product in a short time looks like.

#### What Is a Hackathon?

A hackathon was presented as a fast-paced event where participants build solutions within a limited time, often around 24 to 48 hours. The main goal is not to create a perfect product, but to build a prototype or MVP that clearly demonstrates the core idea.

Main characteristics:

- It takes place within a short period of time.
- It focuses on solving real-world problems.
- It prioritizes prototypes or MVPs.
- It encourages collaboration, creativity, and rapid iteration.
- It gives students a chance to experience pressure similar to real projects.

#### Hackathon Experience

The team shared several real experiences from joining a hackathon, especially activities from the first day:

- Joining partly because of food such as donuts and pizza, but then becoming deeply involved in building the product.
- Attending workshops and the opening session.
- Brainstorming ideas.
- Facing technical problems with a laptop.
- Going through the first day with almost no sleep.

The important lesson is that a hackathon does not only test coding skills. It also tests teamwork, fast decision-making, problem-solving, energy management, and the ability to stay focused under pressure.

#### Lessons Learned from the Hackathon Session

From this session, I learned the following lessons:

- We should not wait for a perfect idea before starting.
- The problem should be clearly defined before choosing technologies.
- Under time pressure, the team must prioritize core features.
- An MVP should prove the main value, not include every possible feature.
- Teamwork and communication are just as important as technical skills.
- Technical problems are normal during product development and can become opportunities to improve problem-solving ability.

### Why We Always Need Confidence

Another session was titled **“Why We Always Need Confidence”**, presented by **Nguyễn Thị Quỳnh Như**. The session focused on the role of confidence in careers, student life, and personal development.

The agenda included:

- Let’s Talk Confidence.
- The Cost of Self-Doubt in Student Life.
- The Science Behind the Struggle.
- The Power of Confidence - Why It’s Essential.
- How to “Hack” Everyday Confidence.
- Wrap-up.

#### Meaning of Confidence

Confidence is not only a personal feeling. It directly affects how students learn, communicate, join activities, attend interviews, and seize opportunities. A lack of confidence can prevent students from asking questions, joining projects, trying new roles, or accurately recognizing their own abilities.

One important point is that confidence is not entirely fixed. It can be developed through small experiences, positive feedback, careful preparation, and consistent action.

#### How to Build Everyday Confidence

From the session, several practical methods can be applied:

- Start with small goals and complete them consistently.
- Prepare carefully before presentations, interviews, or group work.
- Ask questions actively instead of staying silent due to fear of being wrong.
- Treat failure as feedback data, not as proof of personal weakness.
- Join environments such as workshops, hackathons, clubs, or small projects to gain more real experience.

### Current Stack vs AWS Infrastructure Migration

One notable technical topic in the event was the comparison between the **current stack** and the target AWS infrastructure. This part helped me understand how an existing web system can gradually migrate to cloud infrastructure.

#### Current Stack

| Layer | Technology |
| --- | --- |
| Backend/Frontend | Next.js + TypeScript + React |
| Database | MySQL + Prisma ORM |
| Data fetching | SWR |
| Payment gateway | PayOS |
| Email delivery | Resend |
| PDF generation | jsPDF + html2canvas |
| AI / LLM | OpenAI GPT-4.5 |
| Analytics | GA4 + Meta Pixel |
| CI/CD | GitHub Actions |
| Background jobs | Cron jobs |
| Hosting | VPS with Nginx + PM2 |
| Caching | Redis |

#### Target AWS Architecture

| Layer | Technology |
| --- | --- |
| Backend/Frontend | ECS or Amplify, depending on the deployment component |
| Database | Amazon RDS |
| Data fetching | SWR |
| Payment gateway | PayOS |
| Email delivery | Resend |
| PDF generation | jsPDF + html2canvas |
| AI / LLM | Amazon Bedrock |
| Analytics | GA4 + Meta Pixel |
| CI/CD | GitHub Actions |
| Background jobs | CloudWatch / Lambda |
| Hosting | Amplify Hosting |
| Caching | ElastiCache |

This section showed that migrating to AWS does not mean replacing the entire technology stack at once. Some components such as PayOS, Resend, SWR, GA4, and GitHub Actions can remain unchanged. The parts that are most suitable for migration are infrastructure, operations, database, background jobs, caching, and AI services.

#### AWS Architecture Diagram

The architecture diagram presented a web system using components such as:

- **CloudFront** for content delivery and the front access layer.
- **AWS Amplify** for the web application.
- **Amazon Cognito** for user authentication.
- **Amazon RDS** for the database.
- **Amazon S3** for static assets and report PDF storage.
- **AWS Lambda** for report generation tasks.
- **Amazon Bedrock** for AI and report generation.
- **CloudWatch** for logging and metrics.
- External services such as PayOS or Resend that can still be integrated into the architecture.

The key point is that cloud architecture should be designed around a clear data flow: users access the web app, the system authenticates users, processes data, stores results, generates reports, sends report links through email, and monitors the system through logging and metrics.

### The Hidden Iceberg of a Project: DevOps Before Disaster

The final session shown in the images was titled **“The Hidden Iceberg of a Project: DevOps Before Disaster”**, presented by **Trần Minh Quân**. Although the image mainly shows the title slide, the topic suggests an important lesson: the difficult part of a project is often not only the visible interface or features, but also the operational layer behind it.

#### Meaning of the “Hidden Iceberg”

In a software project, users usually see only the interface, features, and final output. However, behind those visible parts are many technical factors that are often underestimated but extremely important:

- Deployment process.
- CI/CD.
- Logging and monitoring.
- Backup and recovery.
- Management of development, staging, and production environments.
- Security and access control.
- Cloud cost control.
- Scalability when traffic increases.
- Incident response when the system fails.

If DevOps is ignored at the beginning, a project may work in the demo phase but become unstable when used in a real environment. Therefore, DevOps should be considered part of system design, not only something to handle at the end.

### What I Learned

#### Knowledge About AWS and Cloud Learning

- Beginners learning AWS often face barriers not only because the technology is difficult, but also because they are afraid of generating unexpected costs during practice.
- AWS Cloud Quest is suitable for building cloud architecture thinking through gamified learning and guided missions.
- Floci helps with quick local testing, reduces cost risks, and is suitable for learning or prototyping.
- A simulated environment cannot completely replace the real AWS environment.
- A reasonable learning roadmap should move from understanding architecture, to local testing, and finally to real deployment.

#### Knowledge About Hackathons and Product Building

- Hackathons help students learn how to build products under time pressure.
- An MVP is more important than trying to build an overly complete product.
- Brainstorming, task division, and team communication are decisive factors.
- Technical incidents are unavoidable, but they are also opportunities to practice problem-solving.
- Joining hackathons helps students gain real experience, improve their portfolios, and build confidence.

#### Knowledge About Confidence

- Confidence directly affects the ability to learn, communicate, and capture opportunities.
- Excessive self-doubt can cause students to miss many growth opportunities.
- Confidence can be trained through small actions, careful preparation, and real experience.
- Communities, workshops, and hackathons are good environments for practicing confidence.

#### Knowledge About AWS Migration and DevOps

- Migration to AWS should be done strategically; it is not necessary to replace every technology at the same time.
- Services such as RDS, S3, Lambda, CloudFront, Cognito, Bedrock, CloudWatch, and ElastiCache can replace or improve many parts of a traditional architecture.
- DevOps is a foundation that helps a project remain stable when deployed in real conditions.
- A good cloud system needs logging, monitoring, CI/CD, security, backup, and cost control.
- If a team only focuses on code and ignores operations, the project may face serious problems when it enters real usage.

### Application to Study and Work

- When learning AWS, I can start with AWS Cloud Quest to understand services and architecture before deploying real infrastructure.
- When working on personal projects, I can use Floci or similar emulation tools to test AWS integration logic locally and avoid unnecessary cloud costs.
- When deploying on real AWS, I need to carefully check resources, set budget alerts, and delete resources after practice.
- When joining hackathons or working on school projects, I should focus on the main problem and build an MVP first.
- When designing systems, I should think early about DevOps factors such as deployment, logging, monitoring, backup, and cost.
- When migrating a system to AWS, I should separate the system into layers such as frontend, backend, database, cache, background jobs, and AI services to choose the most suitable AWS services.
- I also need to build confidence by asking questions, giving presentations, joining workshops, and actively taking responsibility in group projects.
